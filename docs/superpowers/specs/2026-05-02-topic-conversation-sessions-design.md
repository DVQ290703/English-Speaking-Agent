# Topic-Scoped Conversation Sessions — Design Spec

**Date:** 2026-05-02  
**Branch:** TheAnh_fetch_his  
**Status:** Approved

---

## Problem Summary

The chat system has several interconnected bugs and missing features around topic-based conversation sessions:

1. Topic labels display as `topic.undefined.title` / `topic.undefined.desc` — `DashboardPage.jsx` reads `topic.key` but `TOPICS_FLAT` uses `topic.id`.
2. Every message turn creates a new conversation row — `chatRespond()` never sends `conversation_id` to the backend.
3. Frontend sends topic *label* (e.g. `"IELTS Part 1 — Intro"`) instead of topic *code* (e.g. `"ielts_part1"`) — backend `topic_id` lookup always fails.
4. No auto-resume: returning to a topic always starts fresh.
5. Sidebar shows a flat date-grouped list with no topic structure.
6. No way to delete old conversations.
7. No limit on conversations per topic.
8. No client-side caching strategy for switched conversations.

---

## Goals

- Fix all three root-cause bugs (labels, conversation_id, topic code).
- Auto-resume the latest conversation when a user enters a topic.
- Sidebar becomes a hierarchical navigator: categories → topics → conversation history.
- Disconnect = pause (conversation stays `active`).
- New Chat is explicit and blocked at 5 conversations per topic.
- Soft-delete conversations (hidden from UI, retained in DB for future AI use).
- Session naming follows `{topic_title} - Session {N}`.

## Data Layer

### DB Migration

One new column on `conversations`:

```sql
ALTER TABLE conversations ADD COLUMN deleted_at TIMESTAMPTZ DEFAULT NULL;
```

No other schema changes. The `conversations.topic_id` foreign key already exists. Session numbering is computed at query time via `ROW_NUMBER()` — not stored.

### Soft-Delete Rule

All queries visible to the user filter `WHERE deleted_at IS NULL`. Raw data is never removed from the database (retained for AI training).

### Conversation Title Format

Generated by the backend at conversation creation time:

```
{topic_title} - Session {N}
```

Where `N` = count of all conversations (including deleted) for this `user_id + topic_id` + 1. This ensures session numbers never reuse after deletion.

Example: `IELTS Part 1 — Intro - Session 3`

If no topic is associated (free-form chat), title falls back to `"New Conversation"`.

## Backend API

### Fix: `POST /api/chat/respond`

- Accept topic `code` (not label) in the `topic` form field — frontend change aligns with this.
- When `conversation_id` is **absent**:
  1. Count non-deleted conversations for `user_id + topic_id`: `SELECT COUNT(*) FROM conversations WHERE user_id = %s AND topic_id = %s AND deleted_at IS NULL`.
  2. If count ≥ 5: return HTTP 409 `{"detail": "Conversation limit reached", "code": "CONV_LIMIT_REACHED"}`.
  3. Otherwise: create conversation with formatted title (using `ROW_NUMBER` logic for session `N`).
- When `conversation_id` is **present**: existing validation logic is correct, no change.

### New: `GET /api/conversations/for-topic`

Query param: `topic_code` (required).

Returns up to 5 non-deleted conversations for the authenticated user and given topic, latest-first, with computed session numbers.

```json
{
  "topic_code": "ielts_part1",
  "topic_title": "IELTS Part 1 — Intro",
  "conversations": [
    {
      "id": "uuid",
      "title": "IELTS Part 1 — Intro - Session 3",
      "status": "active",
      "session_number": 3,
      "started_at": "2026-05-02T14:30:00Z",
      "updated_at": "2026-05-02T15:10:00Z"
    }
  ],
  "total": 3,
  "limit_reached": false
}
```

SQL pattern for `session_number`:
```sql
SELECT
    c.id::text,
    c.title,
    c.status,
    c.started_at,
    c.updated_at,
    ROW_NUMBER() OVER (
        PARTITION BY c.topic_id ORDER BY c.started_at
    ) AS session_number
FROM conversations c
WHERE c.user_id = %s
  AND c.topic_id = (SELECT id FROM topics WHERE code = %s LIMIT 1)
  AND c.deleted_at IS NULL
ORDER BY c.started_at DESC
LIMIT 5
```

### New: `DELETE /api/conversations/{conversation_id}`

- Ownership check: `WHERE id = %s AND user_id = %s`.
- Sets `deleted_at = NOW(), updated_at = NOW()`.
- Returns HTTP 204 on success, 404 if not found or not owned.

### Updated: `GET /api/conversations`

- Add `AND c.deleted_at IS NULL` to the WHERE clause.
- No change to sort order or limit.
- Note: the sidebar no longer uses this endpoint (replaced by `for-topic`). The existing `useQuery(['conversations'])` call in `VoiceAgent.tsx` should be removed; sidebar data comes from `useQuery(['for-topic', topicCode])` instead.

### Topic Entry Flow (VoiceAgent mount)

When VoiceAgent mounts with `?topic=xxx` in the URL:

```
1. Call GET /api/conversations/for-topic?topic_code=xxx
2. If conversations[0] exists:
     → setCurrentConversationId(conversations[0].id)
     → load messages via GET /conversations/{id}/messages-with-scores
     → populate messages state
3. If empty:
     → currentConversationId stays null
     → new conversation is created on first message send
```

### New Chat Button

Visible when a topic is active. Located near the top of the chat area or sidebar topic-history header.

- If `total < 5`: clear `currentConversationId` and `messages` state. The next message send will create a fresh conversation.
- If `limit_reached === true`: show inline notice — *"You can only keep 5 conversations per topic. Delete one to start a new chat."* Button is disabled.

### Sidebar Redesign (`ConversationSidebar.tsx`)

The sidebar manages two views via local state:

```typescript
type SidebarView = 'browse' | 'topic-history';
interface SidebarState {
  view: SidebarView;
  browseTopic: string | null; // topic code being browsed
  topicConversations: ForTopicResponse | null;
  loadingTopic: boolean;
}
```

**Browse view:**
- Renders `TOPIC_CATEGORIES` as an accordion list (category header → topic rows).
- Currently active topic (from VoiceAgent) is highlighted.
- Clicking a topic:
  1. Calls `GET /api/conversations/for-topic?topic_code=xxx`.
  2. Transitions to topic-history view.

**Topic-history view:**
- Header: `← Back` button (returns to browse) + topic title.
- "New Chat" button at top-right: disabled with tooltip if `limit_reached`.
- Lists up to 5 conversations, each showing:
  - Title (e.g. `IELTS Part 1 — Intro - Session 3`)
  - Relative date (`updated_at`)
  - Trash icon (delete, not shown on currently active conversation)
  - Active conversation highlighted.
- Empty state: *"No history yet — start a new chat."* with "New Chat" button.

**On VoiceAgent mount with active topic:**  
Sidebar initialises directly in topic-history view for that topic (calls `for-topic` on mount).

**Conversation selection (sidebar click):**
1. `setCurrentConversationId(selectedId)`
2. Call `GET /conversations/{id}/messages-with-scores`
3. Populate `messages` state; clear `expandedMsgId`
4. Close sidebar on mobile

**Delete conversation:**
1. Trash icon click → show inline confirm row: *"Delete this session?"* + Confirm / Cancel.
2. On confirm: `DELETE /api/conversations/{id}` → remove from local `topicConversations` list → invalidate `['for-topic', browseTopic]` React Query key.
3. If deleted conversation was the active one: clear `currentConversationId` and `messages`.

### Client-Side Caching

Existing IndexedDB stores (`messages`, `scores`) are keyed by `conversation_id` — no structural change needed.

Cache population: when a conversation is loaded via `messages-with-scores`, write results to IndexedDB under key `conversation_id`. On subsequent loads of the same conversation, read from IndexedDB first; re-fetch from server only if the entry is absent.

On conversation delete: evict the corresponding IndexedDB entries (messages, scores, audio).

## Disconnect Behaviour

Disconnect = pause. No status change on the conversation row. The conversation remains `active`. Returning to the same topic auto-resumes it via the `for-topic` endpoint.

A conversation is only `completed` if explicitly closed in a future feature (out of scope here).

---

## Error States

| Scenario | Behaviour |
|----------|-----------|
| `for-topic` returns 404 / network error | Fall back to null conversation; new conv created on first send |
| 5-limit hit on send (no `conversation_id`) | Backend returns 409; frontend shows sidebar notice |
| Delete fails | Show inline error; keep conversation in list |
| `messages-with-scores` fails on sidebar select | Show error toast; keep previous messages visible |

---

## Out of Scope

- Marking conversations as `completed` (disconnect = pause only)
- Bulk delete all conversations for a topic (can be added later)
- Audio pre-loading / background fetch
- Real-time sync across tabs/devices
